# TEAM_GUIDE

适用范围：`Codex`、`Claude Code` 及其他通用 Agent CLI

本文档面向团队协作，定义在项目中实施 Agent token 优化时的统一原则、角色约定、工作边界与落地方式。

## 1. 目标

团队实施 token 优化的目标不是单纯降低消耗，而是在不明显损伤 agent 能力、任务成功率和交互体验的前提下，降低固定上下文成本与重复加载成本。

团队统一目标如下：

- 降低每个 session 的固定上下文负担
- 降低长会话中的重复上下文负担
- 保持关键技能召回能力
- 保持简单任务的响应速度
- 保持复杂任务的可扩展性

## 2. 基本原则

### 2.1 以项目目录为边界

每个会话默认只服务当前项目目录。

要求：

- 当前目录视为唯一项目根目录
- 默认不读取兄弟目录和无关项目
- 只有在明确需要跨目录引用时，才临时读取外部内容

### 2.2 优先降低固定成本

优先优化这些对象：

- 全局常驻提示
- 项目默认上下文
- 自动加载的历史与知识库
- 技能默认暴露范围

不应优先削弱这些能力：

- 调试
- 测试
- 代码评审
- 规划
- 安全文界

### 2.3 技能可以很多，但默认暴露必须克制

团队允许安装大量 skills，但不应默认把所有 skills 的说明或正文暴露给每个 session。

统一原则：

- 常驻索引少而精
- 正文按需加载
- 长尾技能进入冷层
- 新领域能力通过动态召回机制引入

### 2.4 历史应摘要化，而不是原文续带

默认保留：

- 目标
- 当前状态
- 关键约束
- 已确认决策
- 未决问题
- 下一步

默认不保留：

- 冗长原始对话
- 大段重复日志
- 已失效的旧上下文

## 3. 推荐目录结构

建议每个项目至少具备以下文件：

```text
<PROJECT_ROOT>/
├─ PROJECT_CONTEXT.md
├─ PROJECT_SKILLS.md
├─ SESSION_SUMMARY_TEMPLATE.md
└─ docs/
   └─ token-optimization/
      ├─ token-optimization-report.md
      ├─ token-optimization-summary.md
      └─ token-optimization-checklist.md
```

说明：

- `PROJECT_CONTEXT.md`：项目最小上下文
- `PROJECT_SKILLS.md`：项目默认技能画像
- `SESSION_SUMMARY_TEMPLATE.md`：会话摘要模板
- `docs/token-optimization/`：项目级优化设计与实施文档

## 4. 三层上下文模型

### 4.1 全局最小上下文

全局常驻层只保留：

- 极短系统原则
- 极短安全边界
- 极短技能发现规则
- 少量基础设施技能索引

### 4.2 项目级默认上下文

项目常驻层只保留：

- 项目目标
- 技术栈
- 核心目录
- 当前任务约束
- 当前项目高频 skills

### 4.3 按需加载层与冷层

按需加载层：

- 命中的技能正文
- 命中的阶段规则
- 命中的模板和少量补充材料

冷层：

- 长尾技能
- 长篇知识库
- 旧历史
- 低频模板

## 5. 技能分层规则

### 5.1 基础设施技能

以下类型默认应常驻索引：

- debugging
- tdd
- review
- planning
- documentation

原因：这些是工作流能力，不是领域专用能力。

### 5.2 项目默认技能

每个项目维护自己的默认技能画像，建议控制在 5 到 15 个高频技能内。

### 5.3 长尾冷层技能

长尾技能默认不进入上下文，只在命中时加载，例如：

- 行业专用技能
- 冷门工具链技能
- 低频格式技能
- 低频研究与设计技能

## 6. 项目演化中的新技能召回

团队必须假设：项目会变，任务会变，领域会变。

因此，不能把 skill 使用机制设计成静态白名单。必须引入动态召回机制。

### 6.1 强制触发条件

当出现以下任一情况时，建议触发一次全局 skill 搜索：

- 出现新技术栈、框架或语言
- 出现新文件类型
- 出现新业务领域或新工作流
- 当前方案连续失败
- 当前任务从轻任务升级为调试、研究、评审或复杂规划

### 6.2 晋升规则

新技能命中后，统一按以下方式处理：

- 首次命中：当前任务按需加载
- 短期重复命中：加入项目默认技能索引
- 长期高频命中：晋升为项目常驻能力

## 7. 会话工作方式

### 7.1 旧项目

旧项目统一采用以下流程：

1. 审计当前 token 成本来源
2. 区分固定成本和重复成本
3. 先出文档与优先级
4. 再决定是否改配置、改流程或改项目文件

### 7.2 新项目

新项目统一采用以下流程：

1. 先设计最小上下文结构
2. 先定义项目技能分层
3. 先建立摘要机制
4. 再开始业务实现

### 7.3 跨项目操作

不建议在一个长会话里并行处理多个项目。

推荐做法：

- 一个项目一个会话
- 一个会话一个项目根目录
- 需要跨项目时显式说明

## 8. 统一评估标准

团队判断优化是否有效，必须同时看成本和质量。

### 8.1 成本指标

- 首轮固定上下文 token 是否下降
- 前 5 轮累计 token 是否下降
- 简单任务平均成本是否下降

### 8.2 质量指标

- 首轮技能命中率是否下降
- 首轮澄清次数是否上升
- 任务成功率是否下降
- 返工率是否上升
- 使用者主观体验是否下降

如果成本下降但体验明显变差，应视为失败优化。

## 9. 推荐团队操作规范

建议团队在每个项目中都执行以下动作：

- 创建 `PROJECT_CONTEXT.md`
- 创建 `PROJECT_SKILLS.md`
- 创建 `SESSION_SUMMARY_TEMPLATE.md`
- 创建 `docs/token-optimization/` 文档目录
- 对旧项目做一次 token 成本审计
- 对新项目在启动阶段就完成上下文结构设计

## 10. 团队共识

团队应统一接受以下共识：

- token 优化不是“让 agent 少做事”
- token 优化也不是“把 skills 都删掉”
- token 优化的本质是减少无效上下文、重复上下文和不必要暴露
- 一个健康的 agent 项目应当：
  - 默认轻量
  - 需要时可升级
  - 新领域出现时能动态吸收新能力
