# Agent Token 消耗优化方案摘要

适用范围：`Codex`、`Claude Code` 及其他通用 Agent CLI

## 1. 核心判断

token 过度消耗通常并不来自“安装了很多 skills”，而主要来自：

- 启动时注入过长的系统提示、AGENTS 规则和流程文档。
- 默认暴露了过多技能说明或技能正文。
- 会话历史、知识库、日志原文被重复带入新轮次。
- 同一套重型工作流在轻任务中也被完整加载。
- 多轮对话中反复注入同一批上下文，而不是用摘要替代。

## 2. 优化目标

在不影响每个 session 性能和体验的前提下，降低固定上下文成本与重复加载成本。

必须守住的底线：

- 不降低关键技能召回率。
- 不增加首轮澄清轮次。
- 不显著增加返工率和关键任务失败率。
- 不让简单任务因为复杂流程而变慢。

## 3. 推荐方案

### A. 全局最小上下文

全局常驻层只保留：

- 极短系统原则
- 极短安全边界
- 极短技能发现规则
- 基础设施技能索引

不要把完整流程、完整阶段规则、完整命令说明全部常驻。

### B. 技能分层

- 基础设施技能：常驻索引
  例如 `debugging`、`tdd`、`review`、`planning`、`documentation`

- 项目高频技能：项目级默认可见
  只保留当前项目最常用的 5 到 15 个 skills

- 长尾技能：冷层存放
  默认不进入上下文，仅在命中时按需加载

### C. 索引常驻，正文按需

常驻的是：

- skill 名称
- 一句话描述
- 触发关键词

正文只在命中时读取。

### D. 历史改为结构化摘要

默认只保留：

- 目标
- 当前状态
- 关键约束
- 已确认决策
- 未决问题

不要默认携带完整原始长对话。

### E. 按任务模式加载

- `Lite`：问答、轻修改、轻总结
- `Dev`：编码、调试、测试、评审
- `Research`：检索、规划、多代理

不同模式加载不同层级的规则和技能索引。

### F. 项目演化时动态召回新 skills

项目进行中如果出现新领域、新技术栈、新文件类型或新工作流，应强制触发一次全局 skill 搜索。

新 skill 的处理建议：

- 第一次命中：只在当前任务按需加载
- 重复命中：加入项目级默认索引
- 长期高频：晋升为项目常驻能力

## 4. 最小落地版本

如果只做低风险改造，建议先做五件事：

1. 缩短全局常驻规则文件
2. 为每个项目建立极短 `PROJECT_CONTEXT.md`
3. 建立项目级 skill 白名单
4. 使用 session summary 替代长历史原文
5. 将长尾技能转入冷层，仅命中时加载

## 5. 验收指标

### 成本指标

- 首轮固定上下文 token 下降
- 前 5 轮累计 token 下降
- 简单任务平均成本下降

### 质量指标

- 首轮技能命中率不下降
- 首轮澄清次数不显著上升
- 任务成功率不下降
- 返工率不显著上升

## 6. 结论

最优路线不是“少装 skills”，而是：

- 允许 skills 很多
- 默认少暴露
- 只在命中时加载正文
- 用摘要替代长历史
- 用项目画像和动态召回机制保持能力持续增长
