# Agent Token 优化 Prompt 模板

适用范围：`Codex`、`Claude Code` 及其他通用 Agent CLI

本文件提供三类可直接复用的 prompt：

- 旧项目优化模板
- 新项目初始化模板
- 通用固定指导文本

## 1. 使用说明

建议始终遵循以下原则：

- 当前会话只服务当前项目目录
- 默认不要把其他兄弟目录带入上下文
- 先识别真实的 token 消耗来源，再提出优化方案
- 优化前提是不影响每个 session 的性能和体验

推荐使用顺序：

1. 选定当前项目目录
2. 选择“旧项目优化模板”或“新项目初始化模板”
3. 在 prompt 开头加入“当前目录是唯一项目根目录”的约束
4. 如有需要，再附加“通用固定指导文本”

## 2. 旧项目优化模板

适用场景：

- 已经存在较多文档、规则、skills、历史记录
- 希望在不影响工作流的前提下降低 token 消耗
- 先做审计，再做改造

```text
当前任务只针对项目目录：<PROJECT_PATH>

要求你把 <PROJECT_PATH> 视为当前唯一项目根目录，默认不要读取、分析或修改其他目录，除非我明确要求跨目录引用。

本次任务是为一个“已有项目”设计并落地 token 优化方案。请按以下顺序执行：

1. 先审计当前项目中可能导致 token 过度消耗的来源，包括但不限于：
   - 全局常驻上下文
   - 项目默认上下文
   - skills 暴露范围
   - 历史与知识库携带方式
   - 长文档、长规则、长会话的重复注入
2. 明确区分：
   - 固定上下文成本
   - 重复加载成本
   - 技能暴露成本
   - 历史携带成本
3. 给出一个“不影响每个 session 性能和体验”的优化方案，要求：
   - 不降低关键技能召回率
   - 不显著增加首轮澄清次数
   - 不让简单任务变慢
   - 不显著增加返工率和失败率
4. 方案中必须包含：
   - 全局最小上下文
   - 项目级默认上下文
   - 按需加载层
   - 冷层资源
   - 项目演化中如何动态召回新 skills
5. 先输出评估结论、问题清单、实施优先级
6. 再在项目目录内生成文档，至少包括：
   - PROJECT_CONTEXT.md
   - PROJECT_SKILLS.md
   - token-optimization-report.md
   - token-optimization-checklist.md

本轮默认以分析和文档产出为主，不默认修改项目代码或全局配置，除非我明确要求。
```

## 3. 新项目初始化模板

适用场景：

- 新项目刚开始搭建
- 希望从一开始就控制 token 成本
- 希望把上下文、skills 和 session 机制设计成可扩展结构

```text
当前任务只针对项目目录：<PROJECT_PATH>

要求你把 <PROJECT_PATH> 视为当前唯一项目根目录。现在这是一个“新项目初始化”任务，请直接为该项目设计一套从一开始就节省 token 的 agent 工作结构。

目标：
1. 为新项目建立低 token、可扩展、不中断体验的 agent 上下文架构
2. 从项目初始化阶段就定义：
   - 全局最小上下文
   - 项目级默认上下文
   - skills 分层机制
   - 按需加载机制
   - session summary 机制
   - 项目演化中动态召回新 skills 的机制
3. 约束条件：
   - 不降低关键技能召回率
   - 不增加关键任务失败率
   - 简单任务应保持快速
   - 复杂任务应可平滑升级到重模式
4. 先给出架构草图和设计说明
5. 再生成初始化文档，至少包括：
   - PROJECT_CONTEXT.md
   - PROJECT_SKILLS.md
   - SESSION_SUMMARY_TEMPLATE.md
   - token-optimization-report.md
   - implementation-checklist.md

本轮以项目结构设计和文档初始化为主，不默认创建与业务无关的复杂框架。
```

## 4. 通用固定指导文本

适用场景：

- 任意项目
- 想作为开场固定约束附加到主 prompt 前后
- 用于控制上下文边界和优化方向

```text
请把当前目录视为唯一项目根目录。
默认只读取与当前任务直接相关的文件。
不要把整个仓库或其他兄弟目录自动带入上下文。
请优先识别固定上下文成本、重复加载成本、技能暴露成本和历史携带成本。
请在不降低技能召回率、不增加首轮澄清次数、不让简单任务变慢的前提下进行优化设计。
如果项目出现新技术栈、新文件类型或新任务模式，请设计动态 skill 搜索和晋升机制，而不是依赖静态白名单。
```

## 5. 推荐配套文件

不论是旧项目还是新项目，建议最终都落地以下文件：

- `PROJECT_CONTEXT.md`
- `PROJECT_SKILLS.md`
- `SESSION_SUMMARY_TEMPLATE.md`（新项目建议初始化，旧项目可后补）
- `token-optimization-report.md`
- `implementation-checklist.md`

## 6. 实践建议

### 旧项目

先审计，再收缩常驻上下文，再做技能分层，最后做动态召回。

### 新项目

先设计最小上下文和技能分层，再开始扩展业务文档和工作流。

### 通用原则

- 允许安装很多 skills
- 默认少暴露
- 只在命中时加载正文
- 用摘要替代长历史
- 用项目画像持续吸收新技能
